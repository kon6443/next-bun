Next.js 프론트엔드 변경 코드에 대해 플로우 기반 QA 리뷰를 수행합니다.
모든 판단은 근거 기반(코드 직접 확인)으로만 진행합니다. 추측/추론 금지.
백엔드 프로젝트 경로: `../bun`

## 1단계: 변경 파일 파악

`git diff --name-only`로 변경 파일 목록을 확인합니다.
변경 파일에서 추가/수정된 함수, 컴포넌트, 타입을 식별합니다.

## 2단계: grep 전수 확인

변경된 함수/API/타입 이름으로 **프론트 + 백엔드** 프로젝트 전체 grep 실행.
- 동일 함수가 여러 곳에서 호출되면 중복/불필요 여부 교차 비교
- useEffect 간 중복 패턴, 동일 API 서비스 함수의 다중 호출 경로 주의
- 파일 부분 읽기(offset/limit)만으로 "전체 정상" 판단 금지

## 3단계: 플로우 리스트업

변경 내용을 유저 액션 기준 플로우로 분류합니다.
각 플로우: `유저 액션 → 컴포넌트 핸들러 → API 서비스 호출 → 백엔드 응답 → 로컬 상태 업데이트 → 소켓 이벤트 수신`

## 4단계: 플로우별 코드 추적 검증

각 플로우마다:
1. **중복 API 호출**: 동일 API가 불필요하게 여러 번 호출되지 않는지
2. **소켓 이벤트 핸들러**: self-event 필터링, completedAt 등 필드 반영, 토스트 중복 없는지
3. **타입 일치**: 프론트 타입(task.ts) ↔ API 응답 타입(teamService.ts) ↔ 소켓 페이로드(socket.ts) ↔ 백엔드 DTO
4. **필드 누락**: 새 필드가 타입 → API 응답 → 소켓 핸들러 → UI 렌더링 체인 전체에 포함되는지
5. **낙관적 업데이트**: API 호출 전 UI 반영, 실패 시 롤백 로직 정상 동작
6. **ConfirmModal 사용**: 되돌리기 어려운 액션에 적용, soft delete는 모달 없이 바로 실행

## 5단계: 중복 코드 / 리팩토링

- 동일 패턴이 3곳 이상 반복되면 공통 함수 추출
- 오버엔지니어링 금지

## 6단계: 빌드 검증

- 프론트: `bun run build` 또는 `npx next build`
- 백엔드: `cd ../bun && pnpm run build`
